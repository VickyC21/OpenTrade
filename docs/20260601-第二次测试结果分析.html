结论先行

核心判断：第二次回归的主问题是 timeout 主导，而不是纯粹的功能错误。30 条 timeout 占全部用例 38.0%，却消耗了 97.3% 的总执行时间；这直接把 CLI 的真实体验从“部分命令失败”放大成“整体长时间无响应”。

用户体验层面：auto 当前不是稳定兜底，而是“在少数快失败场景有价值，在长挂起场景几乎无感”。当首候选快速抛异常时，它能给出聚合错误或继续 fallback；当首候选或单个 provider 长时间挂起时，外层 180 秒总超时会吞没一切。

工程层面：确定性源码缺陷并不分散，主要集中在 5 个簇：catalog/enrichment 不一致、stock price live 参数适配、日期参数归一化、watch 首帧输出顺序、脚本判级盲区。这些问题可被较小范围修复，并不需要大改架构。

从非超时样本看，49 条用例里有 36 条达到 pass 或 degraded，有效成功率 73.5%，平均耗时 2.962 秒。这说明项目并不是“全面不可用”；更准确的描述是：可返回的路径里大部分还能工作，但 timeout 与少数适配缺陷把总体感知拖到了很差的水平。

脚本口径的解释边界

scripts/run_incremental_full_regression.py 本身做了很多正确的事：用项目 .venv Python，校验矩阵完整性，持续刷新报告。但它也有明确的观测盲区。

盲区当前行为影响
backend_meta 只从 JSON stdout 提取extract_backend_meta() 只在 stdout 是可解析 dict JSON 时工作（389-403、477-480）auto_stats 会低估真实 fallback；timeout 与非 JSON 输出几乎看不到 candidate 链
failure 分类先匹配网络关键词classify_failure() 在 416-417 行把 502/503/504 视为网络错误stock-holders-latest-count 因日期字符串里包含 20250331 被误记为 network-upstream
pass/degraded 不检查 stdout 语义级警告状态机只看 returncode、stderr、artifact 与 JSON 解析（483-488 行）bond-trades__efinance 第一行已提示代码可能有误，却仍被记成 pass
timeout 没有候选级 trace外层只知道 subprocess 到点被杀（468-471 行）无法判断 auto 卡在第几个候选、provider 内重试了几次、卡在哪个调用

因此第二次 JSON 和结论文档可以回答“哪些 case 没过”，但不能单独回答“auto 兜底到底有没有价值”或“180 秒到底耗在哪一层”。这部分必须结合源码链路一起看。

脚本本身的优点不能忽略：它在开头就明确写明“使用项目虚拟环境调用真实 CLI 与真实第三方 API，覆盖全部叶子命令、全部支持 backend 和代表性运行时模式”，并在 45-46 行把 CLI 前缀固定到 .venv\Scripts\python.exe；validate_case_matrix() 还会校验命令覆盖、backend 覆盖、参数覆盖和 runtime tag 覆盖（299-347 行）。也就是说，第二次回归暴露出的系统性问题，基本不是“矩阵过窄”导致的假象。

根因分组

1. 串行 failover 与 provider 统一重试叠加，放大 timeout

AUTO_CANDIDATE_ORDER 在 opentrade/backends/resolver.py:13-17 固定为 AKSHARE -> YFINANCE -> EFINANCE。CommandFacade._invoke_auto() 在 opentrade/facade.py:65-85 串行逐个尝试候选；每个 non-side-effect handler 又会在 BackendProvider.execute() 中进入 provider 级统一重试（opentrade/backends/base.py:99-157），底层重试器是 Retry(max_retries=8, delay=True)（opentrade/retry_utils.py:38）。这意味着一个候选只要长时间不返回，就会独占掉几乎全部 180 秒预算，后续候选根本没有展示机会。

resolver.py:13-17 facade.py:65-85 base.py:99-157 retry_utils.py:38

search__auto 是最明确的证据：stderr 持续输出长进度条，说明 provider 已经深入执行，并非路由层直接失败；但外层 180 秒一到就被 subprocess 杀掉，最终只剩一个 timeout 结果。

2. auto 的真实 fallback 被统计低估，但也确实不够强

stock-price-latest__auto 明确记录了 candidate_chain=["yfinance", "efinance"]、final_backend="efinance"，说明真实 fallback 成功发生过。与此同时，auto_stats 只有 1 次 fallback 成功，并不代表 auto 总体价值只有这一例，而是脚本只能在“命令返回且 stdout 是 JSON”时观察到它。

不过这不是给 auto 洗白。第二次回归里 auto 共 11 条，只拿到 1 条 pass、2 条 fail、8 条 timeout，实际用户感受仍然很差。更准确的说法是：auto 机制本身不是零价值，但它的观测口径和 timeout 行为都让价值显著打折。

3. catalog 与 enrichment/observation 回补链不一致，导致确定性源码缺陷

bond.profile 的失败不是上游接口波动，而是 CLI 自己在 observation 路径上把命令图谱接错了。默认非 raw 视图会在 CommandExecutor.invoke() 中进入 enrich_market_data() 和 build_observation_output()（opentrade/executor.py:29-39），opentrade/observation.py:345-364 随后调用 fetch_standard_history_for_request()。问题在于它为 bond.profile 解析出了 bond.price.history，但 fetch_standard_history_for_request() 却硬调用 get_shared_command_definition()（opentrade/enrichment/service.py:160-169），而 bond.price.history 并不在 SHARED_COMMANDS 中，只在 single-backend 集合里，最终触发 KeyError: 未知共享命令: bond.price.history（opentrade/command_catalog.py:238-315）。

这类问题的性质很明确：provider 主调用可能已经成功，真正坏掉的是共享增强链路。所以它属于可快速修复的产品层 defect，而不是“上游不可控”。

4. stock price live 同时踩中了参数适配错误和 failover 截断

参考 catalog 里，stock price live 的 fs 参数被定义为 multiple=true 且默认空列表（.skill/opentrade/references/command-catalog.json:623-637）；schema 构建后，coerce_schema_value() 会把它归一化成 Python 列表（opentrade/request_schema.py:95-102）。但 efinance 通用 handler 直接把 request_data 原样透传给上游函数（opentrade/backends/providers.py:109-124），于是 stock-price-live__efinance 最终把 ['m:0+t:6'] 传给了 efinance.stock.get_realtime_quotes，上游明确报错：KeyError: 指定的行情参数 ['m:0+t:6'] 不正确。

这还不是全部。stock price live 的 supported backends 是 efinance 和 akshare（opentrade/command_catalog.py:80-81），套入全局 auto 顺序后，auto 候选链实际变成 AKSHARE -> EFINANCE。akshare handler 又在 opentrade/backends/providers.py:304-325 明确要求市场必须是 A_stock，测试值 m:0+t:6 会触发 ValueError。而 is_failover_eligible_error() 把 ValueError 视为不可 failover（opentrade/facade.py:26-33），所以 stock-price-live__auto 在第一跳就被截断，根本不会继续尝试 efinance。

结论不是“stock price live 两个 backend 都不行”，而是：一个 backend 参数不兼容，另一个 backend 候选被 failover 策略提前截断。

5. 日期参数缺少统一归一化，脚本还把其中一例误判成网络问题

stock-holders-latest-count__efinance 的真实异常是 ValueError: time data '20250331' does not match format '%Y-%m-%d'；stock-leaderboard-daily__efinance 同样是 ValueError: time data '20250530' does not match format '%Y-%m-%d'。这说明 CLI 层传入的是紧凑日期串，但 provider 或上游函数期望横杠格式日期。它们都属于确定性适配问题。

脚本又在 classify_failure() 里把 503 当成网络错误关键词（scripts/run_incremental_full_regression.py:416-417），所以 20250331 这例被误归到 network-upstream。这让问题看起来像随机网络波动，实际上不是。

6. 上游网络问题真实存在，但第三方解析脆弱把波动放大成硬失败

stock-profile__efinance 先出现东财请求被远端断开，随后三方库在空或缺字段结果上继续执行，最终落到 KeyError: ['股票代码']。stock-holders-top10__efinance 则在 27 秒后因 requests.exceptions.JSONDecodeError 失败。bond-flow-today__efinance stdout 先打印 证券代码 "113519" 可能有误，随后上游 JSONPath 访问触发 TypeError: 'bool' object is not subscriptable。这些都说明：部分失败不是 CLI 自己路由错了，而是第三方库在异常响应、空响应或半结构化响应上的防守太弱。

futures-price-history__efinance 是相对合理的例子：stdout 仍输出 {}，stderr 记录东财网络异常，脚本将其记为 degraded 而不是 fail，这比直接炸栈更接近真实体验。

7. watch 的空白 45 秒是设计结果，不是偶发

create_watch_command() 只是把子命令包装成带 --watch 的调用（opentrade/commands.py:302-316）。真正的循环逻辑在 CommandExecutor._run_watch()，但它先调用 result = self.invoke(request)，之后才 click.echo(header)（opentrade/executor.py:49-69）。因此如果第一次调用阻塞，用户在整个阻塞窗口里什么都看不到。watch__none 的 45 秒完全无 stdout/stderr，正是这个设计的直接体现。

8. 存在一簇“无输出 180 秒超时”的静默挂死，需要补 trace 才能继续定位

market-price-live__efinance、futures-catalog__efinance、futures-price-live__efinance、futures-trades__efinance、多条 yfinance 和 auto 的 history/profile 命令都在 180 秒超时且 stdout/stderr 为空。仅凭第二次回归材料，能确认它们“没有在超时前完成”，但还不能确认是 provider 内部无超时网络调用、第三方库卡死、线程池等待，还是 observation 渲染链阻塞。这个簇的下一步必须是补候选级、provider 级 trace，而不是继续拍脑袋猜。

对回测脚本的评估

scripts/run_incremental_full_regression.py 并不是一份随便拼起来的 smoke script。它有三个很有价值的工程特征：

  1. 它明确固定使用项目虚拟环境解释器：VENV_PYTHON = PROJECT_ROOT / ".venv" / "Scripts" / "python.exe"，CLI 前缀也直接绑定到这一路径（45-46 行）。
  2. 它在 docstring 与实现上都强调“全量矩阵 + 增量落盘”：脚本头部说明每条用例完成后都会刷新 JSON 与 HTML 报告，validate_case_matrix() 还会检查命令覆盖、backend 覆盖、业务参数覆盖和 runtime tag 覆盖（299-347 行）。
  3. 它会主动剥离本机代理环境变量（359-364 行），所以这批失败不能简单甩锅给本机代理配置。

但作为“可解释的全量真实回归”脚本，它还有四块需要尽快补强：

问题当前行为建议
auto 观测过窄只有 stdout 是 JSON dict 时才能抽出 backend_meta无论输出格式如何，都在 CLI trace 或 stderr 里补 requested/resolved/final backend 与候选级耗时
failure 分类误判字符串级匹配把日期中的 503 误当网络错误先识别明确的 Python 异常类型，再做关键词兜底；数字匹配要带边界
pass 过宽stdout 里的警告文本不会影响状态把“代码可能有误”“空结果但无 stderr”等语义加入 degraded 规则
timeout 不可定位只知道 subprocess 到 180 秒或 45 秒被杀把 provider 级 trace、每个 candidate 的开始、结束、异常写入报告

如果只修业务代码不修这份脚本，下一轮回归仍然会在“真实问题已变少，但报告解释力不够”上吃亏。

P0bond.profile catalog/enrichment 不一致确定性源码缺陷，修复成本低、收益直接回补链改用 get_command_definition() 或显式兼容 single-backend history 命令
P0stock price live 适配错误同一条用户命令在 efinance、akshare、auto 三路全失败把 fs 归一化为单值字符串，或为 live 场景单独 schema；同时评估是否应把“backend 不支持当前 market”视为可 failover
P0日期参数归一化两个失败是稳定可复现的纯适配问题在 CLI/provider 入口统一把 YYYYMMDD 转成 YYYY-MM-DD 或按上游要求分流
P1脚本 failure 分类与 pass 判级会把确定性 defect 写成网络波动，也会把 warning 结果写成纯 pass改进 classify_failure() 与状态机，增加 stdout warning 识别
P1watch 首帧无输出用户感知为“命令卡死”，排障成本高先输出 header 或 starting line，再执行第一次 invoke；必要时加 spinner 或 phase trace
P1第三方解析器脆弱网络波动后直接炸成 KeyError 或 JSONDecodeError在 provider 标准化前先做空响应、非 JSON、缺字段保护
P2timeout 簇的精细归因目前证据不足，容易误判下一轮带 trace 重跑 market/futures/yfinance 静默超时簇，再决定是否需要 provider 专项限时或线程池调整
